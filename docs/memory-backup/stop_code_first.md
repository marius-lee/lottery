# 🚨 禁止直接动手改代码

## 正确流程

1. **分析** — 仔细思考问题，弄清楚现状
2. **出方案** — 写出清晰的解决方案，说明改什么、为什么这样改
3. **讨论** — 提交方案给用户审查，用户可能修改方案
4. **确认** — 双方都认可方案后
5. **动手** — 再开始写代码

## 错误示范（我经常犯的）

用户提出问题 → 我直接开始 Edit/Write 改代码 → 用户不满意改动 → 浪费时间

## 为什么

用户是系统架构总监，需要看到完整的方案设计。跳过讨论环节是不专业、不尊重用户的行为。参照 [[think_before_coding]]。

**How to apply:** 每次收到修改需求，先用文字输出分析和方案，等用户确认后再动代码。
